# 图书自动借书机系统 - 架构设计方案

## 一、项目概述

### 1.1 项目背景
图书自动借书机系统是一个面向图书馆的自助借阅服务系统，旨在提升图书借阅效率、降低人工成本、提升读者体验。本项目作为课程大作业，重点展示**微服务架构设计**和**现代技术栈的整合应用**。

### 1.2 核心业务场景
- **自助借书**：读者通过机器扫描图书条码或输入ISBN，完成借阅登记
- **自助还书**：读者归还图书，系统自动计算逾期罚款
- **图书检索**：支持书名、作者、关键词的全文检索
- **借阅查询**：查看个人借阅记录、应还日期
- **管理后台**：图书管理、用户管理、数据统计

### 1.3 架构目标
- **高可用**：服务无单点故障，支持水平扩展
- **高性能**：热点数据缓存、高并发场景下稳定运行
- **可扩展**：微服务架构，便于功能模块迭代
- **可观测**：完善的监控告警体系
- **易部署**：容器化部署，环境一致性

## 二、技术选型方案

### 2.1 技术栈总览

| 层级 | 技术组件 | 版本 | 核心作用 |
|------|----------|------|----------|
| **前端层** | Vue 3 | 3.x | 前端框架，构建借书机界面和管理后台 |
| **前端层** | ECharts | 5.x | 数据可视化，借阅统计图表展示 |
| **网关层** | Nginx | 1.27+ | 反向代理、静态资源服务、API网关 |
| **服务治理** | Nacos | 2.x | 服务注册发现、配置中心 |
| **服务治理** | Sentinel | 1.8+ | 限流、熔断、降级 |
| **消息中间件** | RabbitMQ | 3.x | 异步通知、流程解耦、削峰填谷 |
| **缓存层** | Redis | 7.x | 热点数据缓存、分布式锁、Session存储 |
| **检索层** | Elasticsearch | 8.x | 图书信息全文检索 |
| **监控层** | Prometheus | 2.x+ | 指标采集、监控告警 |
| **安全层** | Spring Security | 6.x | 认证授权、权限控制 |
| **容器层** | Docker | - | 应用容器化 |
| **编排层** | Kubernetes | 1.24+ | 容器编排、弹性扩缩容 |

### 2.2 选型理由说明

#### 2.2.1 Spring Security vs Shiro
| 维度 | Spring Security | Shiro |
|------|-----------------|-------|
| **社区活跃度** | Spring官方维护，更新活跃 | Apache社区，更新较慢 |
| **功能丰富度** | OAuth2、JWT、LDAP等全面支持 | 基础功能完善，高级功能需扩展 |
| **与Spring整合** | 深度整合，注解驱动 | 需手动配置 |
| **学习曲线** | 配置复杂，概念多 | 简单直观，上手快 |
| **企业应用** | Spring生态首选，市场占比高 | 中小企业仍有使用 |

**选型结论**：选用 **Spring Security**。理由：
1. 与Spring Boot微服务架构天然契合
2. 功能更全面，适合展示架构能力
3. 企业应用主流选型，就业竞争力强

#### 2.2.2 Nginx 作为网关
- **反向代理**：统一入口，隐藏内部服务细节
- **静态资源服务**：直接服务Vue前端静态文件
- **负载均衡**：支持多实例流量分发（作为能力点展示）
- **SSL终止**：证书配置，HTTPS支持

#### 2.2.3 Nacos 服务治理
- **服务注册发现**：微服务动态注册，客户端负载均衡
- **配置中心**：配置集中管理，支持热更新
- **配置对比**：
  | 配置中心 | 优点 | 缺点 |
  |----------|------|------|
  | Nacos | 集成注册发现，AP+CP切换，可视化界面 | 资源消耗较大 |
  | Apollo | 配置管理专业，版本控制完善 | 需独立部署，较重 |
  | Config | Spring生态，与Git集成 | 需配合Bus实现热更新 |

**选型结论**：Nacos。一体化解决方案，适合课程项目快速搭建。

#### 2.2.4 Sentinel 流量控制
- **限流**：QPS限流、并发线程数限流
- **熔断**：慢调用比例、异常比例、异常数熔断
- **降级**： fallback 机制，服务容错
- **对比Hystrix**：
  | 特性 | Sentinel | Hystrix |
  |------|----------|---------|
  | 隔离策略 | 信号量/线程池 | 线程池为主 |
  | 熔断策略 | 慢调用/异常比例/异常数 | 异常比例 |
  | 实时监控 | 内置控制台 | 需配合Turbine |
  | 注解支持 | 完善 | 完善 |
  | 维护状态 | 阿里活跃维护 | 停止新功能 |

**选型结论**：Sentinel。功能更丰富，控制台直观，适合演示。

#### 2.2.5 RabbitMQ 消息队列
| 特性 | 说明 |
|------|------|
| **异步通知** | 借阅成功后异步发送短信/邮件通知 |
| **流程解耦** | 借书流程与通知、统计流程解耦 |
| **削峰填谷** | 借阅高峰时缓冲请求，保护数据库 |
| **失败重试** | 消息确认机制，保证最终一致性 |

**选型理由**：
- 成熟稳定，企业应用广泛
- 多种消息模型（Direct、Topic、Fanout）
- 管理界面直观，便于演示

#### 2.2.6 Redis 综合应用
| 应用场景 | 说明 |
|----------|------|
| **热点缓存** | 热门图书列表、图书详情缓存，降低DB压力 |
| **分布式锁** | 借书流程并发控制，防止超借 |
| **Session存储** | 分布式Session，支持多实例部署 |
| **计数器** | 借阅量统计、限流计数器 |
| **排行榜** | 热门图书借阅排行榜（ZSet） |

**缓存策略**：
- 缓存 Key 命名：`books:list:{params}`、`books:detail:{id}`
- 过期时间：列表60秒，详情120秒
- 缓存更新：写操作后主动删除缓存（Cache-Aside模式）

#### 2.2.7 Elasticsearch 全文检索
| 特性 | 说明 |
|------|------|
| **全文检索** | 书名、作者、摘要、关键词分词检索 |
| **相关性排序** | TF-IDF算法，相关度高的结果优先 |
| **高亮显示** | 搜索关键词高亮 |
| **聚合统计** | 按分类、出版年等维度聚合 |

**索引设计**：
```json
{
  "mappings": {
    "properties": {
      "id": { "type": "long" },
      "title": { "type": "text", "analyzer": "ik_max_word" },
      "author": { "type": "text", "analyzer": "ik_max_word" },
      "summary": { "type": "text", "analyzer": "ik_max_word" },
      "category": { "type": "keyword" },
      "publishDate": { "type": "integer" },
      "status": { "type": "keyword" }
    }
  }
}
```

#### 2.2.8 Prometheus 监控体系
| 监控维度 | 指标示例 | 说明 |
|----------|----------|------|
| **应用层** | `http_requests_total` | HTTP请求总量 |
| **应用层** | `http_request_duration_seconds` | 请求延迟分布 |
| **应用层** | `borrow_operations_total` | 借阅操作计数 |
| **中间件** | `redis_connected_clients` | Redis连接数 |
| **中间件** | `rabbitmq_queue_messages` | MQ队列堆积 |
| **系统层** | `node_cpu_seconds_total` | CPU使用率 |
| **系统层** | `node_memory_used_bytes` | 内存使用量 |

**告警规则**：
- QPS超过阈值（如1000）告警
- 错误率超过5%告警
- MQ队列堆积超过1000条告警
- 服务实例下线告警

#### 2.2.9 Docker & Kubernetes
| 层级 | 作用 |
|------|------|
| **Docker** | 应用容器化，环境一致性，镜像版本管理 |
| **Kubernetes** | 容器编排，服务发现，负载均衡，弹性扩缩容 |

**K8s资源设计**：
- Deployment：无状态服务（借阅服务、用户服务）
- StatefulSet：有状态服务（MySQL、Redis、ES）
- Service：服务暴露，负载均衡
- Ingress：入口路由（替代Nginx部分功能）
- ConfigMap/Secret：配置管理
- HPA：水平自动扩缩容

#### 4.1.3 关键技术点

**1. Redis 分布式锁**
```java
// 伪代码示例
public boolean borrowBook(Long bookId, Long userId) {
    String lockKey = "lock:borrow:book:" + bookId;
    String lockValue = UUID.randomUUID().toString();
    
    // 尝试获取锁
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, lockValue, 5, TimeUnit.SECONDS);
    
    if (!locked) {
        throw new BusinessException("当前借阅请求较多，请稍后重试");
    }
    
    try {
        // 执行借书逻辑
        return doBorrow(bookId, userId);
    } finally {
        // 释放锁（使用Lua脚本保证原子性）
        String script = "if redis.call('get', KEYS[1]) == ARGV[1] " +
                       "then return redis.call('del', KEYS[1]) else return 0 end";
        redisTemplate.execute(new DefaultRedisScript<>(script, Long.class),
            Collections.singletonList(lockKey), lockValue);
    }
}
```

**2. Sentinel 限流配置**
```java
@SentinelResource(
    value = "borrowBook",
    blockHandler = "borrowBlockHandler",
    fallback = "borrowFallback"
)
public BorrowResult borrowBook(Long bookId, Long userId) {
    // 业务逻辑
}

// 限流触发时的处理
public BorrowResult borrowBlockHandler(Long bookId, Long userId, BlockException ex) {
    return BorrowResult.fail("系统繁忙，请稍后再试");
}

// 熔断降级处理
public BorrowResult borrowFallback(Long bookId, Long userId, Throwable ex) {
    return BorrowResult.fail("服务暂时不可用");
}
```

**限流规则配置**：
| 规则类型 | 阈值 | 说明 |
|----------|------|------|
| QPS限流 | 100 | 单机每秒最多100次借书请求 |
| 并发线程数 | 50 | 同时处理的借书请求不超过50个 |
| 熔断策略(慢调用) | 1s | 响应超过1秒视为慢调用 |
| 熔断比例 | 50% | 慢调用比例超过50%触发熔断 |

### 10.3 容错设计

#### 10.3.1 熔断降级策略

| 场景 | 触发条件 | 降级策略 | 恢复策略 |
|------|----------|----------|----------|
| 借书服务 | 慢调用比例 > 50% | 返回"系统繁忙"提示 | 等待5分钟后尝试恢复 |
| 还书服务 | 异常比例 > 30% | 记录失败，后台任务重试 | 监控异常率下降后恢复 |
| 搜索服务 | 响应时间 > 2s | 降级为数据库查询 | ES恢复后切回 |
| 通知服务 | 消费者下线 | 消息持久化，等待消费者恢复 | 消费者上线后自动消费 |

#### 10.3.2 重试机制

```java
@Retryable(
    value = {SQLException.class, IOException.class},
    maxAttempts = 3,
    backoff = @Backoff(delay = 1000, multiplier = 2)
)
public void processBorrow(Long borrowId) {
    // 业务逻辑
    // 第一次失败后等待1秒重试
    // 第二次失败后等待2秒重试
    // 第三次失败后等待4秒重试
    // 都失败则抛出异常
}

@Recover
public void recover(SQLException e, Long borrowId) {
    // 重试全部失败后的兜底处理
    // 记录失败日志，标记为待人工处理
    log.error("Borrow processing failed after retries: {}", borrowId, e);
    markAsFailed(borrowId);
}
```
