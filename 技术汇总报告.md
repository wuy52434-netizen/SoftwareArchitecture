# 图书自动借书机系统 - 技术汇总报告

## 一、项目定位

本项目是课程大作业中的图书自动借书机系统。答辩时的重点不是读者门户，而是：

1. 自助借书机主流程。
2. 自助还书主流程。
3. 管理后台的统计和管理辅助展示。

系统默认入口是 `http://localhost`，地址栏保持根路径并直接渲染借书机首页，不再跳到 `/portal`。

## 四、核心接口和业务逻辑

### 4.1 借书相关

| 接口 | 作用 | 说明 |
|---|---|---|
| `GET /api/books/scan?code=...` | 扫码查询图书 | 可输入 ISBN 或条码 |
| `GET /api/books/copy/{copyId}` | 查副本 | 按 `copy_id` 定位实体副本 |
| `GET /api/books/copy/barcode/{barcode}` | 按条码查副本 | 借书机扫码主接口 |
| `GET /api/books/{id}/available-copy` | 查可借副本 | ISBN 查询时选一个可借副本 |
| `POST /api/borrow` | 借书 | 传 `bookId`、`copyId`、`userId` |

借书核心代码在 `backend/borrow-service/src/main/java/com/library/borrow/service/BorrowService.java`。它做了四件事：

1. 用 Redis 锁锁住 `bookId:copyId`。
2. 调用用户服务检查用户状态和借阅上限。
3. 调用图书服务把副本状态从 `available` 改成 `borrowed`。
4. 写入 `borrow_record`，然后发 RabbitMQ 事件。

### 4.2 还书相关

| 接口 | 作用 | 说明 |
|---|---|---|
| `POST /api/return` | 还书 | 支持按 `borrowId` 或 `bookId` |
| `GET /api/borrow-records/{id}` | 查单条借阅记录 | 后台和借书机都能用 |
| `GET /api/my-borrows` | 查我的借阅 | 读者自查 |

还书核心逻辑：

1. 找到借阅记录。
2. 计算是否逾期。
3. 恢复副本状态和库存。
4. 更新借阅记录。
5. 发 RabbitMQ 还书事件。

### 4.3 搜索和统计

| 接口 | 作用 |
|---|---|
| `GET /api/search` | Elasticsearch 图书检索 |
| `GET /api/search/hot` | 热门图书 |
| `GET /api/stats/dashboard` | 管理后台统计图数据 |
| `POST /api/stats/borrow` | 统计借阅事件 |
| `POST /api/stats/return` | 统计归还事件 |

## 五、数据结构为什么这样设计

### 5.1 表结构

| 表 | 关键字段 | 说明 |
|---|---|---|
| `book_info` | `id`, `isbn`, `title`, `available_copies` | 书的主信息 |
| `book_copy` | `copy_id`, `book_id`, `barcode`, `status` | 实体副本 |
| `borrow_record` | `record_id`, `book_id`, `copy_id`, `user_id`, `return_date` | 借阅记录 |

### 5.2 为什么必须有 `copy_id`

ISBN 只是“这本书是什么”，`copy_id` 才是“借出去的是哪一册”。
借书机演示如果只看 ISBN，就没法解释“同一本书的某一册已经借出，另一册还能借”的情况。
所以本项目把借书主键设计成 `book_id + copy_id`，条码 `barcode` 作为机器扫码入口。

## 六、两个可用性场景

### 场景 1：热门书高并发借阅

- 问题：多人同时借一本热门图书。
- 风险：库存被重复扣减。
- 方案：Sentinel 限流 + Redis 锁 + MySQL 条件更新。
- 结果：最多只会有一个请求成功。

### 场景 2：网络抖动 / 连续点击重复借书

- 问题：借书机按钮被连续点击，或网络重发请求。
- 风险：重复借出同一副本。
- 方案：前端 loading 防重复点击 + `copy_id` 唯一副本 + Redis 锁 + 副本状态原子更新。
- 结果：第一次成功，第二次返回“图书副本不可用”。

## 七、架构说明

### 开发架构

- 后端：Spring Boot 多模块微服务。
- 前端：Vue 3 + Vite + Pinia + Element Plus。
- 公共模块：统一响应、异常、JWT、Redis 工具。

### 物理架构

- 一台答辩机器。
- Docker Compose 拉起所有服务和中间件。
- 通过 Nginx 暴露唯一前端入口。

### 运行架构

`浏览器 -> Nginx -> Gateway -> 业务服务 -> MySQL/Redis/RabbitMQ/ES`

### 逻辑架构

- 接入层：Nginx、Gateway
- 安全层：Spring Security、JWT
- 业务层：user/book/borrow/search/notify/stats
- 数据层：MySQL、Redis、ES、RabbitMQ

### 数据架构

- MySQL：权威业务数据
- Redis：缓存、锁、计数
- ES：搜索读模型
- RabbitMQ：异步事件

### 8.2 JMeter 演示方案

JMeter 作为演示工具更权威，因为它能模拟真实并发、网络重发和连续点击。演示请求走 `http://localhost/api/borrow`，即完整经过 Nginx、API Gateway、Nacos 服务发现、Sentinel、Redis 锁和后端业务服务。

JMeter 文件位置：

| 文件 | 场景 | 线程数 | 目的 |
|---|---|---:|---|
| `demo/jmeter/borrow-concurrency.jmx` | 热门书高并发借阅 | 20 | 证明同一副本不会被超借 |
| `demo/jmeter/duplicate-click.jmx` | 网络抖动/连续点击重复借书 | 5 | 证明重复提交不会重复借出 |
| `demo/jmeter/reset-demo-copy.sql` | 演示前数据复位 | - | 恢复 `BC0024001` 为可借状态 |

演示前先复位数据：

```powershell
cd "E:\Ai Code\SoftwareArchitecture"
Get-Content demo\jmeter\reset-demo-copy.sql | docker exec -i library-mysql mysql --default-character-set=utf8mb4 -uroot -proot123 library
```

确认目标副本：

```powershell
docker exec library-mysql mysql --default-character-set=utf8mb4 -uroot -proot123 library -e "SELECT bc.copy_id, bc.book_id, bc.barcode, bc.status, bi.title, bi.available_copies FROM book_copy bc JOIN book_info bi ON bi.id=bc.book_id WHERE bc.barcode='BC0024001';"
```

当前演示库里 `user1` 的 `user_id` 是 `11`，JMeter 文件默认使用 `userId=11`。如果重建数据库后 ID 变化，先执行：

```powershell
docker exec library-mysql mysql --default-character-set=utf8mb4 -uroot -proot123 library -e "SELECT user_id, username, status FROM user WHERE username='user1';"
```

再把 JMeter `Demo variables` 中的 `userId` 改成查询结果。

GUI 演示步骤：

1. 打开 JMeter。
2. File -> Open，选择 `demo/jmeter/borrow-concurrency.jmx`。
3. 点击启动，查看 `View Results Tree` 和 `Aggregate Report`。
4. 重新执行 reset SQL。
5. 打开 `demo/jmeter/duplicate-click.jmx`，再次点击启动。
6. 展示响应体中的 `code=200` 和 `code=2007`。

命令行演示：

```powershell
& "E:\apache-jmeter-5.6.3\bin\jmeter.bat" -n -t demo\jmeter\borrow-concurrency.jmx -l demo\jmeter\borrow-concurrency.jtl -e -o demo\jmeter\report-concurrency
Get-Content demo\jmeter\reset-demo-copy.sql | docker exec -i library-mysql mysql --default-character-set=utf8mb4 -uroot -proot123 library
& "E:\apache-jmeter-5.6.3\bin\jmeter.bat" -n -t demo\jmeter\duplicate-click.jmx -l demo\jmeter\duplicate-click.jtl -e -o demo\jmeter\report-duplicate
```

注意：JMeter 计划里加入了 `$.code == 200` 的业务断言，因此后续被系统拦截的请求会在 Aggregate Report 中显示为失败。这里的“失败”不是系统异常，而是证明重复借书被架构拦截。

### 8.3 两个 JMeter 场景的预期结果

| 场景 | JMeter 观察点 | 预期结果 | 架构解释 |
|---|---|---|---|
| 热门书高并发借阅 | 20 个线程同时请求同一个 `bookId=24, copyId=1` | 只有 1 个业务成功，其余被锁冲突、副本不可用或系统繁忙拦截 | Redis 锁串行化请求，MySQL 条件更新副本状态兜底 |
| 网络抖动/连续点击重复借书 | 5 个线程模拟同一用户连续提交同一副本 | 第一次 `code=200`，后续常见 `code=2007 图书副本不可用` | `book_copy.status` 已从 `available` 变成 `borrowed`，重复请求不能再次成功 |

演示后查询数据库：

```powershell
docker exec library-mysql mysql --default-character-set=utf8mb4 -uroot -proot123 library -e "SELECT copy_id, book_id, barcode, status FROM book_copy WHERE barcode='BC0024001'; SELECT record_id, user_id, book_id, copy_id, status, return_date FROM borrow_record WHERE copy_id=1 ORDER BY record_id DESC LIMIT 5;"
```

讲解口径：

> 我们不是靠手点页面证明可用性，而是用 JMeter 模拟真实并发和网络重复提交。即使多个请求同时进入，Redis 分布式锁和 MySQL 副本状态条件更新也能保证同一实体副本只被借出一次。
