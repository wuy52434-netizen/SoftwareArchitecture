# 图书自动借书机系统 - 四人团队开发方案

---

## 一、团队协作概览

### 1.1 项目目标
基于 **12个新技术** 设计并实现一个图书自动借书机系统，完成课程大作业并准备答辩。

### 1.2 核心原则
| 原则 | 说明 |
|------|------|
| **技术均衡** | 每人负责3个新技术，均衡分配 |
| **职责清晰** | 角色明确，避免推诿 |
| **协作高效** | 定期同步，问题及时解决 |
| **文档规范** | 所有产出文档化，便于答辩 |

---

## 二、技术分配方案（每人3个技术）

### 2.1 技术分组原则

根据**技术关联性**和**开发流程顺序**进行分组：

| 分组维度 | 技术列表 | 说明 |
|----------|----------|------|
| **前端层** | Vue 3、ECharts | 前端展示和可视化 |
| **网关层** | Nginx、Spring Security | 入口和安全 |
| **服务治理** | Nacos、Sentinel | 服务管理和流量控制 |
| **异步消息** | RabbitMQ、Prometheus | 异步处理和监控 |
| **数据层** | Redis、Elasticsearch | 缓存和检索 |
| **容器层** | Docker、Kubernetes | 容器化和编排 |

Closes #12

### 8.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|

| 测试时间不足 | 中 | 中 | 提前准备测试用例，边开发边测试 |

### 8.2 简化方案（Backup Plan）

如果某些技术实现困难，可采用以下简化方案：

| 技术 | 简化方案 | 说明 |
|------|----------|------|
| **Kubernetes** | 仅用 Docker Compose | K8s作为"了解"，不实际部署 |
| **Sentinel** | 简化限流规则 | 仅实现QPS限流，不做复杂熔断 |
| **Prometheus** | 简化监控 | 仅收集关键指标，Grafana用预置模板 |
| **Elasticsearch** | 数据库模糊查询 | 如果ES配置困难，先用MySQL LIKE |

## 十、总结

### 10.1 核心原则回顾

1. **技术均衡**：每人3-4个技术，难度适中
2. **责任清晰**：前端/安全/服务治理/数据容器，四大模块
3. **时间可控**：4周计划，预留缓冲
4. **答辩导向**：每个技术都准备"是什么、为什么、怎么用"

### 10.2 成功关键

| 关键因素 | 说明 |
|----------|------|
| **尽早开始** | 不要等，先搭框架，再填充细节 |
| **频繁沟通** | 不要等到问题大了才说，每天站会 |
| **文档先行** | 先写设计文档，再写代码 |
| **准备备选** | 每个技术点都要有简化方案 |
| **模拟答辩** | 提前演练，发现问题 |
